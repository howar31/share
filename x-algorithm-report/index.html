原始碼技術解析報告

針對 xAI 於 2026 年 5 月公開的 xai-org/x-algorithm 程式庫,五個平行 agent 深入探勘五大子系統後彙整的完整技術解析。

總覽

專案簡介

這個程式庫是 X(原 Twitter)「For You」資訊流背後的核心推薦系統。它的工作是:從你追蹤的帳號(in-network)與全站貼文語料庫(out-of-network)中,檢索、排序、過濾貼文,最終產生你看到的那串時間軸。

這次釋出最重要的一句話 —— 演算法的哲學轉向:

「我們已經消除了系統中每一個手工設計的特徵(hand-engineered feature)與大部分的啟發式規則(heuristics)。由一個基於 Grok 的 transformer 模型扛起所有重活 —— 透過理解你的互動歷史(按讚、回覆、分享…)來判斷什麼內容跟你相關。」

換句話說:過去的推薦系統靠數百個人工特徵(發文者粉絲數、貼文新鮮度、互動率…)。新系統把這些幾乎全丟掉,改由一個 transformer 直接「閱讀」你的行為序列來決定相關性。這大幅降低了資料管線與服務基礎設施的複雜度。

系統架構:五大子系統

整個程式庫由五個各自獨立的子系統組成,分屬兩種語言 —— 線上服務層用 Rust(追求低延遲),機器學習層用 Python / JAX。

Home Mixer Rust

編排層。組裝整個 For You 資訊流,串接所有階段,對外提供 gRPC 端點。

Candidate Pipeline Rust

可重用框架。定義 Source / Hydrator / Filter / Scorer / Selector 等 trait,是 Home Mixer 的骨架。

Thunder Rust

記憶體內貼文倉儲。即時消化 Kafka 事件,提供「追蹤帳號貼文」的次毫秒查詢。

Phoenix Python · JAX

ML 核心。兩塔檢索模型 + 基於 Grok 的排序 transformer,預測互動機率。

Grox Python

內容理解服務。用 LLM 對貼文做垃圾偵測、安全政策分類、品質評分與向量嵌入。

一次請求的完整生命週期

Home Mixer 採用兩層堆疊管線。外層 ForYouCandidatePipeline 負責「資訊流組裝」(貼文 + 廣告 + 推薦追蹤);內層 PhoenixCandidatePipeline 負責「貼文檢索與排序」這件重活。內層之所以獨立,是因為它對外暴露成 ScoredPostsService gRPC 端點,可被其他服務單獨呼叫。

階段內層 PhoenixCandidatePipeline 做的事
Query Hydrators
(15 個)		抓取使用者情境:互動行為序列、追蹤名單、封鎖/靜音名單、訂閱、人口統計、追蹤的主題與 Starter Pack、曝光 bloom filter、IP 地理位置等。
Sources
(6 個,平行)		從 Thunder(站內)、TweetMixer、Phoenix 檢索、Phoenix 主題、Phoenix MoE 取候選;若 Redis 有快取則走 CachedPostsSource 捷徑。
Hydrators
(10 個,平行)		補上核心貼文資料、引用貼文、影片長度、媒體旗標、訂閱、作者資訊、語言碼、主題等。
Filters
(14 個,循序)		濾掉重複、過舊、自己的貼文、轉推去重、訂閱牆、已看過、已服務、靜音關鍵字、社交圖封鎖等。
Scorers
(3 個,循序)		PhoenixScorer(ML 預測)→ RankingScorer(加權合成 + 多樣性)→ VMRanker(可選的價值模型重排)。

(平行,非阻塞)		寫 Redis 候選快取、發 Kafka 重排日誌、記錄已服務歷史等。

外層管線拿到排序好的貼文後,透過 BlenderSelector 把廣告、「推薦追蹤」模組、提示卡片混入適當位置,最後序列化成 URT thrift 格式回傳給客戶端。

五大子系統詳解

1. Home Mixer Rust — 編排層

角色:整個 For You 資訊流的總指揮。它執行上述兩層管線,並對外提供兩個 gRPC 端點:ScoredPostsService(回傳純排序貼文,供下游使用)與 ForYouFeedService(回傳組裝完成的資訊流項目)。

技術棧
Rust + Tokio 非同步;gRPC 用 Tonic(mTLS、gzip/zstd 壓縮);protobuf / thrift 序列化。
快取
moka 行程內 LRU + xds_redis 分散式 Redis(zstd level 6 壓縮)。
實驗框架
xai_feature_switches(依使用者/國家/客戶端比對)+ xai_decider(執行期布林旗標)。
外部依賴
Gizmoduck(使用者資料)、TweetEntityService、SocialGraph、Strato、Phoenix、Thunder、AdIndex、VisibilityFiltering 等 20+ 服務。

候選來源(Candidate Sources)

來源說明
ThunderSource站內貼文 —— 來自你追蹤帳號的近期貼文。
PhoenixSource站外主力檢索 —— Phoenix 嵌入向量 ANN 檢索叢集。
PhoenixTopicsSource主題過濾的站外檢索,用特定主題 entity ID。
PhoenixMOESource來自獨立的 Phoenix Mixture-of-Experts 叢集。
TweetMixerSource另一路站外推薦(HOME_RECOMMENDED_TWEETS)。
CachedPostsSourceRedis 命中時直接回傳已評分快取,跳過全部檢索/評分。

廣告混入(Ads Module)

2026 年 5 月新增的 home-mixer/ads/ 模組負責把廣告插入資訊流。兩種策略由 feature switch 切換:

  • SafeGapAdsBlender —— 找出「安全間隙」:兩則連續的非中度風險貼文之間的位置,把廣告塞進去。
  • PartitionOrganicAdsBlender —— 把安全的自然貼文均分成數段,每段貢獻一組「(上方貼文, 廣告, 下方貼文)」三明治。

兩種策略都套用品牌安全三條捨棄規則:若廣告相鄰的貼文是低風險/被列入特定作者 handle/含封鎖關鍵字,該廣告就被丟棄。貼文的品牌安全等級(Safe / LowRisk / MediumRisk)由 SafetyLabelStore 的標籤集計算 —— 值得注意的是,未經 Grok 評分或未經 PTOS 審查的貼文,預設視為 MediumRisk。

2. Candidate Pipeline Rust — 可重用框架

角色:一個泛型的 Rust 框架(對 query 型別 Q 與候選型別 C 泛型化),把推薦管線拆成一組可組合的 trait。Home Mixer 就是用它搭起來的。

Trait職責執行方式
QueryHydrator檢索前先豐富 query(抓使用者特徵、社交圖)全部平行
Source從一個檢索後端取一批候選全部平行
Hydrator批次豐富候選(補作者、媒體資料)全部平行
Filter把候選分成 kept / removed 兩堆循序
Scorer對候選打分數循序(每個都改完才換下一個)
Selector依分數排序,選 Top-K單一,同步
SideEffect事後副作用(記錄、快取暖機)tokio::spawn 背景執行,不阻塞回應

設計亮點 —— 全程「優雅降級」(graceful degradation):

框架沒有任何 fail-fast。任何一個 hydrator/source/scorer 失敗,只會被記錄 error! 然後靜默跳過 —— 該來源貢獻零候選,或該欄位不更新。一個元件壞掉只會降低品質,永遠不會讓整個管線中斷。此外 Hydrator 契約明文禁止丟棄候選(要丟請用 Filter 階段),框架以「回傳長度必須等於輸入長度」來強制這點。

每個元件都有 enable(&query) 方法 —— 這是 per-request 功能旗標與 A/B 實驗的主要機制。所有階段都用 tracing span 加 xai_stats_receiver 指標做觀測。

3. Thunder Rust — 站內貼文倉儲

角色:一個記憶體內的貼文倉儲 + 即時消化管線,專門服務「站內候選」—— 也就是你追蹤帳號的貼文。它消化 Kafka 的貼文建立/刪除事件,在 RAM 維護每位使用者的索引,提供次毫秒級查詢,完全不打外部資料庫。

記憶體資料模型

採用兩層指標結構來省記憶體:

  • TinyPost(僅 16 bytes:post_id + created_at)—— 存在每位使用者的時間軸佇列裡。
  • LightPost(完整貼文資料)—— 存在全域的 posts map,只有當貼文真的通過所有過濾條件時才會被取出。

每位使用者的貼文分三類存放:original_posts(原創)、secondary_posts(回覆與轉推)、video_posts(含影片)。五張 map 全部用 DashMap(分片鎖,無全域鎖),讓消化與服務可以高度平行。

Kafka 消化的雙管線

一個 is_serving 旗標決定 Thunder 實例扮演哪種角色:

  • Feeder 模式(v1) —— 消化原始的 Thrift TweetEvent 洪流,正規化後重新發布成較輕量的 protobuf InNetworkEvent。
  • Serving 模式(v2) —— 消化已預處理的 protobuf 事件,直接寫入記憶體倉儲,並對外提供 gRPC 服務。

這種角色分離把「原始事件解碼」與「低延遲服務」解耦。保留期預設 2 天,背景每 2 分鐘自動修剪一次過舊貼文。

效能關鍵細節:查詢迴圈、排序、修剪全都跑在 tokio::task::spawn_blocking,避免 CPU 密集工作餓死非同步 I/O 執行緒。讀取 DashMap 時還有一個刻意的「複製後再做巢狀查詢」模式 —— 先把值複製出來、立刻釋放分片讀鎖,再做下一次查詢,以防死鎖。每個查詢還有 per-request 超時,在追蹤名單很大時提供有界延遲。

4. Phoenix Python · JAX — ML 核心

角色:整套演算法的機器學習大腦,用 Python + JAX / DeepMind Haiku 寫成(沒有用 PyTorch)。它做兩件事:檢索(retrieval)與排序(ranking)。

① 檢索:兩塔模型(Two-Tower)

負責從全站語料庫快速撈出站外候選:

  • User Tower —— 把使用者特徵與互動歷史編碼成一個向量;會經過共用的 Phoenix transformer,再對所有有效位置做 mean-pooling 並 L2 正規化。
  • Candidate Tower —— 把每則貼文編碼成向量(預設走兩層 SiLU MLP 投影)。
  • 相似度搜尋 —— 兩邊都 L2 正規化後做點積(等同 cosine similarity),用 jax.lax.top_k 取 Top-K。釋出的範例語料庫含 53.7 萬則預先算好的運動類貼文向量。

② 排序:帶「候選隔離」的 Transformer

排序模型 PhoenixModel 包著一個從 Grok-1 開源版移植來的 transformer(grok.py;pyproject.toml 的專案名稱直接就叫 grok-1)。輸入序列的排版是:

[使用者 token ×1] + [互動歷史 ×127] + [候選貼文 ×64]  →  共 192 個 token

最關鍵的架構設計 —— 候選隔離注意力遮罩(Candidate-Isolation Attention Mask):

從一個因果(下三角)遮罩出發,然後把「候選對候選」的整個子區塊歸零,只保留對角線(每則候選對自己的 self-attention)。

結果:每則候選都能雙向看到「使用者 token + 完整歷史」,但看不到其他任何候選。這讓每則貼文的分數成為 (使用者情境, 該候選) 的獨立函數 —— 分數不會因為這一批裡有哪些其他貼文而改變,因此一致、可快取。這正是 README「Key Design Decisions」第 2 點的程式碼實現。

Hash-Based Embeddings(雜湊嵌入)

檢索與排序都不用傳統的「每個 ID 一個 embedding」查找表。每個實體(使用者、貼文、作者)用 N 個獨立的線性同餘雜湊函數映射到共用嵌入表的不同列:

raw = (id × scale[j] + bias[j]) % modulus
bucket = (raw % (vocab_size − 1)) + 1   # 0 保留給 padding

得到的 N 個 embedding 再經線性投影合成。這讓固定大小的嵌入表能涵蓋無上限的 ID 空間。

多行為預測(Multi-Action Prediction)

模型不是只輸出單一「相關性」分數,而是對每則候選預測 19 種行為機率:

favorite 讚reply 回覆repost 轉發quote 引用 click 點擊profile_clickvideo_viewphoto_expand share 分享share_via_dmshare_via_copy_linkdwell 停留 quoted_clickfollow_author 追蹤作者 not_interested 不感興趣 block_author 封鎖 mute_author 靜音 report 檢舉 dwell_time

注意最後四項是負向行為 —— 模型也學著預測「你會討厭什麼」。

5. Grox Python — 內容理解服務

角色:X 的即時內容理解服務,2026 年 5 月新增。它坐在「原始貼文流」與「推薦管線」之間 —— 每則新貼文一發出,就透過 Kafka 進來,跑過一整套 LLM 驅動的分類器與嵌入模型,把結構化標註(垃圾標籤、安全政策旗標、品質分數、主題分類、向量嵌入)寫回平台資料庫,供下游排序演算法使用。

任務執行引擎

由三個長駐元件組成,各跑在獨立 OS 行程,透過共享佇列通訊:

  • Dispatcher —— 從 Kafka 拉任務塞進佇列;讀回結果、ack offset、失敗重試;以 max_in_flight 做反壓。
  • Engine —— 從佇列取任務,每個任務開一個 asyncio 協程,交給 PlanMaster.exec() 處理。
  • gRPC Server —— 與前兩者並行啟動。

Plan 是一張任務 DAG(有向無環圖),帶 REQUIRED_ELIGIBILITY 閘門 —— 同一個 payload 可帶多個 eligibility,因此 PlanMaster 能在同一則貼文上平行跑多個 plan,避免重複載入資料。

LLM 分類器

分類器偵測什麼
SpamEapiLowFollowerClassifier低粉絲帳號發出的垃圾回覆。
BangerInitialScreenClassifier貼文品質 / 爆紅潛力;輸出 quality_score、主題分類、內容標籤,以及 slop_score(1–3,標記低品質/AI 灌水內容)。
PostSafetyDeluxeClassifier熱門貼文的安全旗標(NSFW、血腥、暴力等布林標註)。
SafetyPtosCategoryClassifierPTOS 兩階段政策管線的第一階段:辨識違反了哪些政策大類(暴力媒體、成人內容、垃圾、仇恨/霸凌、暴力言論、自殘等)。
SafetyPtosPolicyClassifierPTOS 第二階段:對每個被標記的大類做細粒度政策判定。Deluxe 模式會用 Grok 4.2 推理模型。
ReplyScorer為回覆排序評分(0–3 分品質)。

Grox 如何餵養推薦演算法

Grox 的輸出分兩路流向下游:

  • 安全 / 執法寫入 —— 垃圾標籤、PTOS 安全標籤、品質分數、主題分類寫進 Strato 的 UnifiedPostAnnotations / SafetyPostAnnotations。排序演算法讀這些來提升、降權或過濾貼文(回想前面 Home Mixer 的品牌安全:「未經 Grok 評分」的貼文預設 MediumRisk —— 這個 Grok 評分就是 Grox 產出的)。
  • 嵌入寫入 —— V3/V4/V5 多模態貼文嵌入向量寫進 Strato,供基於內容的近鄰檢索使用。

值得一提的設計:ASR(語音轉文字)管線跑在多個獨立子行程,用 ffmpeg 從影片抽音軌,避免緩慢的轉錄拖垮主任務迴圈;每個子行程還註冊週期性 gc.collect() 以對抗大量圖片/音訊位元組造成的記憶體壓力。

關鍵機制深入

評分與排序

排序由內層管線的三個 scorer 循序完成:

第一步 — PhoenixScorer:取得 ML 預測

呼叫 Phoenix 模型,對每則候選回傳 19+ 種行為機率(favorite_score、reply_score… report_score)。對新使用者(行為歷史少於門檻)會切換到專屬的推論叢集。

第二步 — RankingScorer:加權合成最終分數

把多種機率合成一個分數。核心公式:

最終分數 = Σ ( weighti × P(actioni) )

正向行為(讚、轉發、分享、追蹤作者)權重為正;負向行為(封鎖、靜音、檢舉、不感興趣)權重為負,把使用者大概會討厭的內容往下壓。關鍵實作細節:

  • 所有權重都是 feature-switch 參數,每次請求即時載入 —— 可在不重新部署的情況下調整線上權重。
  • 影片權重(VQV)若影片長度低於門檻會被歸零。
  • 作者多樣性懲罰 —— 依分數排序後,同一作者的後續貼文分數會被一個衰減乘數壓低:(1 − floor) × decay^位次 + floor,確保資訊流不被單一作者洗版。
  • 站外(OON)權重 —— 站外候選的最終分數會再乘上一個 OON 權重因子(主題請求、新使用者各有不同因子)。

第三步 — VMRanker:可選的價值模型重排

啟用時,把所有候選送進 VMRanker 服務做最終重排,並可套用 DPP(行列式點過程)多樣性參數。

過濾機制

過濾分兩階段執行 —— 評分前濾掉「不該進入排序」的內容,選擇後再做最終可見度把關:

評分前過濾(Pre-Scoring)

  • DropDuplicatesFilter — 重複貼文 ID
  • AgeFilter — 超過時效門檻的舊貼文
  • SelfTweetFilter — 使用者自己的貼文
  • RetweetDeduplicationFilter — 同內容轉推去重
  • IneligibleSubscriptionFilter — 無權存取的付費牆內容
  • PreviouslySeen / ServedPostsFilter — 已看過 / 已服務過
  • MutedKeywordFilter — 含使用者靜音關鍵字
  • AuthorSocialgraphFilter — 封鎖 / 靜音的作者
  • TopicIdsFilter — 主題不符

選擇後過濾(Post-Selection)

  • VFFilter — 可見度過濾:移除已刪除 / 垃圾 / 暴力 / 血腥等貼文
  • AncillaryVFFilter — 移除附屬貼文(被丟棄的引用 / 祖先貼文)
  • DedupConversationFilter — 同一對話串的多個分支去重,只留最高分的那則

五個核心設計決策

README 明列五點,對照程式碼後可印證:

  1. 沒有手工特徵 —— 系統完全依賴 Grok transformer 從使用者互動序列學習相關性,不做任何人工特徵工程,大幅降低資料管線複雜度。
  2. 排序時的候選隔離 —— transformer 推論時,候選之間互相看不見、只能看使用者情境;確保分數不受同批其他貼文影響,因此一致且可快取。
  3. 雜湊式嵌入 —— 檢索與排序都用多個雜湊函數做嵌入查找,讓固定大小的表涵蓋無限 ID 空間。
  4. 多行為預測 —— 不預測單一「相關性」,而是預測許多種行為的機率(含負向行為)。
  5. 可組合的管線架構 —— candidate-pipeline crate 把「管線執行/監控」與「業務邏輯」分離,獨立階段平行執行、優雅處理錯誤,新增 source/filter/scorer 都很容易。

結語

重點整理

  • 哲學:「廢除手工特徵,讓一個基於 Grok 的 transformer 透過讀你的行為序列來決定相關性」是這套演算法相較舊版最大的轉變。
  • 雙來源:站內(Thunder,你追蹤的人)+ 站外(Phoenix 兩塔檢索,全站語料庫),合併後共同排序。
  • 排序核心:一個帶「候選隔離注意力遮罩」的 transformer,對每則貼文獨立預測 19 種行為機率,再加權合成 —— 正向加分、負向扣分。
  • 分層清楚:線上低延遲服務(Home Mixer / Thunder / Candidate Pipeline)用 Rust;機器學習(Phoenix / Grox)用 Python+JAX。
  • 工程務實:全程優雅降級(任何元件壞掉不中斷管線)、feature-switch 驅動可即時調權重、Redis 候選快取捷徑、廣告插入帶品牌安全把關。
  • 內容治理:新增的 Grox 服務用 LLM 對每則貼文做垃圾偵測、安全政策(PTOS)分類與品質評分,結果回寫並影響排序與廣告相鄰安全。
  • 透明度的界線:釋出的是「具代表性」的程式碼與一個迷你預訓練模型;實際線上模型的特定擴展優化(可能含 MoE)、真實權重數值與訓練資料並未公開。
